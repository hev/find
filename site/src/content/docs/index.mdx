**hev ask** is an ask-first search overlay for [Astro](https://astro.build)
documentation sites. You add one integration, drop one component in your layout,
and readers get a `⌘K` command palette over your content collection that leans
toward **answering the question they actually have**:

- **Ask a question** — type more than one word (or press Enter) and Claude
  decides its own sub-queries, navigates your knowledge graph, then **streams a
  grounded answer** with inline deep links to the sections it drew from. The
  overlay even **suggests questions** to ask, drawn from your docs.
- **Instant keyword lookup** — type a single term and you get keyword results as
  you type, ranked with the same knowledge graph and linked to the exact heading
  (`/docs/page#section`), with no API key in the path.

Readers who never want the model call can flip to keyword-only and it stays out
of the way. This site is built with hev ask and searches itself — press `⌘K`
and try *"how does the knowledge graph stay fresh"* to watch the answer stream
in over the real corpus you're reading.

## Who this is for

You're building or maintaining an Astro 5 docs site. Your content lives in a
[content collection](https://docs.astro.build/en/guides/content-collections/)
as Markdown or MDX. You want search that:

- works out of the box without standing up a service or running a crawler,
- deep-links to the right section instead of dumping the reader at the top of a
  long page, and
- can answer a question phrased in the reader's words, not just match keywords.

If you only need keyword search over a static site and never want an API key in
the loop, [Pagefind](https://pagefind.app) is simpler and a great fit — see
[Tradeoffs](/docs/tradeoffs) for an honest comparison.

## How it fits an Astro site

hev ask ships as a single package, `@hev/ask`, exposing an Astro integration
as its default export:

- The **integration** injects an on-demand `/api/ask` route, exposes resolved
  config and the knowledge graph through virtual modules, and runs the
  knowledge-graph build during `astro build`.
- The **`SearchOverlay.astro`** component renders the palette. Any element with
  a `data-hev-ask-open` attribute opens it; `⌘K` is wired automatically.
- The **`/api/ask`** endpoint answers queries — keyword mode with no key,
  agentic mode when `ANTHROPIC_API_KEY` is set — and serves the suggested
  questions the overlay shows on open.
- The knowledge graph is built **offline** — most conveniently with a
  **Claude Code skill** (no API key, no per-build token spend), or with the
  **`hev-ask-kg`** CLI as a fallback — and verified with the same CLI.

<Callout label="the corpus is your collection">
hev ask only searches the Astro content collection(s) you configure. There's
no external index and nothing to keep in sync — the docs you render *are* the
search corpus.
</Callout>
