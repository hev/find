**hev ask** distills a docs site — [Astro](https://astro.build), Docusaurus,
VitePress, MkDocs, or any folder of markdown — into an **ask digest**: a compact
**directory**, one small markdown file per section, a distilled mirror of your
docs that an agent navigates the way it navigates any directory, and that a
`⌘K` overlay answers your readers from.

The digest is **host-neutral**: it's built from your markdown, not your
renderer. Astro gets a turnkey integration; every other framework gets the same
overlay as a one-script drop-in. Build the digest **offline** with the bundled
skill (and your Claude Code subscription), commit the tree, and your docs become
readable three ways from one artifact. Perfect for technical documentation,
internal wikis, and other medium-sized corpora.

## One artifact, three surfaces

The digest is built **inside your coding agent** — the bundled Claude Code
skill writes the `.hev-ask/` tree using your existing **agentic coding
subscription**, so there's no `ANTHROPIC_API_KEY` and no per-build token spend.
Commit the tree, and the same files serve:

- **the `⌘K` overlay**, for humans — instant keyword results as readers type,
  plus a grounded Claude answer on Enter, every result deep-linked to the exact
  heading;
- **the `ask` CLI**, for agents — `tree`, `ls`, `head`, `cat`, `facts`, `grep`
  over path keys, keyless, from any shell;
- **the `ask mcp` server**, for agents — one tool that hydrates the whole tree
  to local disk, after which the agent reads it with the file tools it
  already has.

The overlay *synthesizes* for a human; the CLI and MCP hand an agent the
*files* and let its own tools do the rest.

<Callout label="the corpus is your content">
hev ask only reads the content you point it at — an Astro content collection, a
Docusaurus `docs/` tree, a set of markdown globs. There's no crawler, no
external index, and nothing to keep in sync — the docs you render *are* the
digest, and the committed `.hev-ask/` tree is reviewable per section in every PR.
</Callout>

## Who this is for

You're building or maintaining a docs site whose content lives in Markdown or
MDX — on **Astro** (the turnkey integration), **Docusaurus**, **VitePress**,
**MkDocs**, or anything that renders a folder of markdown. You want search and
answers that:

- work out of the box without standing up a service or running a crawler,
- deep-link to the right section instead of dumping the reader at the top of a
  long page,
- can answer a question phrased in the reader's words, not just match keywords, and
- are **queryable by your coding agent**, not only by humans in a browser.

On Astro that's one integration; on every other framework it's a static overlay
you drop in, plus an optional hosted endpoint for the agentic answers. See
[Frameworks](/docs/frameworks) for the path on yours.

If you only need keyword search over a static site and never want an API key in
the loop, [Pagefind](https://pagefind.app) is simpler and a great fit — see
[Tradeoffs](/docs/tradeoffs) for an honest comparison.
